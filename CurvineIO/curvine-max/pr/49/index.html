HashBG Read Cache

CurvineIO/curvine-max · PR #49 · 设计解读 · 回到 PR

背景

在分布式文件系统里，读性能往往比写更难优化：数据可能躺在远端 UFS（底层统一存储）上，每次读都要跨网络拉取。缓存是经典解法——在靠近计算的 worker 节点上放一份副本，命中则快，未命中则回源。难点在于：缓存必须是可丢弃的，绝不能与权威元数据纠缠；并发写入同一逻辑块时不能互相踩踏；删除与覆盖必须在 reader 安全窗口之后回收磁盘。

本设计在 Curvine worker 上定义了完整的 HashBG Read Cache 第一阶段：从拓扑放置、RPC 协议、worker 分层存储，到 client 侧文件级 prewarm/read/delete，再到后台 GC 与容量分账。下面先从广到窄建立上下文。

分布式文件系统入门（可跳过）

Curvine 把文件切成固定大小的 block，由 worker 节点在本地磁盘（或 SPDK）上持有物理副本。Client 通过 RPC 与 worker 通信：open → 流式 write/read → commit/close。Master 维护文件元数据（inode、长度、版本），worker 维护块级物理布局。

系统组件（简化）
Client ⇄ RPC ⇄ Worker (dev1) · Worker (dev2) · Worker (dev3)
UFS（权威存储） ← 未命中时回源 → Client 读路径

示例：文件 /data/model.bin inode=42，block_size=64MB，offset 0 对应 block_index=1

Block Group (BG) 一组 worker 的集合，负责同一批逻辑块的副本。HashBG 用 hash(inode_id, block_index) % bg_count 决定某个逻辑块属于哪个 BG——注意不含文件版本，这样同一逻辑块的不同缓存版本始终落在同一组 worker 上。

与本 PR 直接相关

此前，worker 只有「普通 block」一种本地数据模型：BlockStore 直接管理 block_map 与 VFS 路径。HashBG read cache 是第二类数据——key 不同、可丢弃、有独立 RPC、需要 metadata-first 删除和后台回收。

本设计定义了三件事的大框架：

  1. Worker 分层重构：BlockStore 变为 facade，下挂 RegularBlockStore 与 HashBgBlockStore，共享 PhysicalDataset。
  2. 端到端读缓存流：client 文件级 planner/reader/writer + worker hash RPC handler + RocksDB catalog + 物理文件布局。
  3. 运维闭环：retire-reclaim + orphan-sweep 后台任务；HashBG 容量分账与 LRU 淘汰钩子。

直觉

可以把 HashBG read cache 想象成「按内容寻址的、可丢的 CDN 边缘节点」——但它服从文件系统的 block 切分与 BG 副本规则，而不是 HTTP URL。

放置与 key：版本不进 BG 哈希

考虑玩具文件：inode=100，读 offset=0，block_size=1MB，当前 f_version=3（由 mtime/len 派生）。

Key 分解（ReadCache purpose）
  • bg_id = hash(100, block_index=1) % 32 → 假设 = 7（与 f_version 无关）
  • block_id = LogicalBlockId { inode=100, block_index=1, f_version=3 }
  • block_version = 0（read cache 阶段固定）

磁盘路径示例：
…/hashbg/bg_7/i_100/b_1/v_3/attempt-uuid.tmp → rename → …/v_3/<object_id>.blk

为什么 bg_id 不含 f_version？ 若版本参与 BG 哈希，reader 在只知道 inode 和 block_index 时无法确定该问哪组 worker——而 f_version 又嵌在 block_id 里用于区分缓存版本。把 BG 选择与版本解耦，消除了循环依赖。

写路径：UUID 尝试对象 + 指针切换

两个 prewarm 任务同时写同一 logical block 时，若共享确定性 temp 文件名，会互相 unlink、发布短写对象。解法是：每次写分配唯一 object_id（UUID），字节写入独立 temp 文件，长度校验通过后 rename 为不可变 ready 对象，最后原子切换 catalog 指针（last-commit-wins）。输家的物理文件进入 retire 队列，由后台 grace 期后 unlink。

一次 HashBG 写（玩具数据：expected_len=1048576）
open_for_write → reserve 账本 + 创建 .tmp → 流式 write 1MB → commit: rename + publish_ready

覆盖写：catalog 先指向新 object_id，旧 object 入 retire 队列——请求路径不直接 unlink。

Phase 1 边界 HashBG 只支持 file-backed data dir（DISK/SSD/HDD/MEM），显式拒绝 SPDK_DISK。普通 Curvine block 的 SPDK 路径不受影响。

读路径：BG 副本重试 → UFS fallback

Client 的 HashFileReader 把文件 read range 切成若干 logical block，对每个 block 先走 HashBG：按 BG worker 列表尝试读，结构化错误（CacheMiss、NotReady、Corrupt）在副本间 retry；全部 exhausted 后对该 block fallback 到 UFS 权威读。Worker 读路径不做同步 fill——miss 就是 miss，由上层决定是否 prewarm。

读一个 block 的决策（f_version=3, bg_id=7）
HashBlockReader → dev2? 命中 ✓
HashBlockReader → dev1 miss → dev3 miss UfsReader 回源

文件级统计：cache_blocks vs fallback_blocks 写入 HashFileReadResult。

GC 与容量分账

WorkerBackground 注册两个互补任务：

  • hash-bg-retire-reclaim：消费持久化 retire 队列，grace 期后 unlink + 释放 hash_final 账本。
  • hash-bg-orphan-sweep：扫描 hashbg/ 目录，清理 crash/race 漏网的 orphan（慢速兜底）。

容量设计第一版：

  • VfsDir 增加 hash_tmp_bytes / hash_final_bytes 双桶；fs_used() 四桶求和，heartbeat 自动看到 HashBG 占用。
  • 写 open 时 reserve_hash_space + hash_budget 准入；commit/abort/Drop 对称 release。
  • 启动 recover_hash_accounting() 扫描 catalog 重建 hash_final。
  • LRU 淘汰钩子（catalog touch + 水位触发）——部分逻辑在容量 PR 落地。

open: hash_tmp += 1MB → commit: tmp−1MB, final+=1MB → retire+unlink: final−1MB
