Summary

This PR lays the foundation for HashBG (Hash Block Group) — a hash-based block routing system for the Curvine distributed filesystem. It introduces the core contracts that future PRs (P2–P6) will build upon: logical block identity, block-group placement, cluster topology view, and proto-level RPC definitions for hash block write/read/delete operations.

Simultaneously, the worker's monolithic storage layer (VfsDataset) is split into a physical layer (PhysicalDataset) and a logical catalog (RegularBlockCatalog), orchestrated by RegularBlockStore. This separation is critical: future HashBG block types (read cache, write buffer) will share the same physical dataset but maintain their own logical indexes.

How It Works

The HashBG system divides the cluster into a fixed number of Block Groups (BGs). Each logical block is deterministically mapped to a BG via a stable hash, and each BG has a set of worker replicas with a designated primary.

→

→

Why split PhysicalDataset from the catalog?
Read-cache blocks and write-buffer blocks will live on the same disks as regular blocks. Sharing one Arc<PhysicalDataset> avoids double-counting capacity and lets each block-purpose orchestrator manage its own logical index independently. The physical layer works entirely on &self — placement chooser and SPDK allocator use internal locks.
Watch out for: BlockMeta not yet split
BlockMeta still bundles path rules, reader/writer factories, and capacity state with the catalog entry. Splitting it into a lean PhysicalBlockRecord is deferred until the HashBG model is validated (per the layering plan). This means the catalog entries are heavier than they should be — but the interface is ready for the split.
Key design decision: stable hashing with splitmix64
splitmix64(inode_id) ^ splitmix64(block_index) gives good distribution without a dependency on a hashing crate. The hash is deterministic — the same block always maps to the same BG, enabling consistent lookup across client and master.

Code Walkthrough

1. Proto & RPC Contracts (curvine-common/proto/)

common.proto adds LogicalBlockIdProto (inode_id, block_index, f_version). worker.proto adds four new message types: HashBlockWriteRequest/Response, HashBlockReadRequest/Response, HashBlockDeleteRequest/Response. rpc_code.rs reserves RPC codes 84–86 for these operations.

2. Topology & Routing (curvine-common/src/topology/ — NEW)

cluster_view.rs defines the entire routing vocabulary:

  • LogicalBlockId — 16-byte identity (inode_id + block_index + f_version), with as_u128()/from_u128() for compact storage
  • BlockPurpose enum — Regular, ReadCache, WriteBuffer
  • HashBlockKey — full identity tuple (purpose + bg_id + block_id + version)
  • BgInfo — a block group: id, purpose, storage type, worker list, primary index
  • ClusterView — the routing table: bg_count BGs, each mapped to workers. Implements HashBgRouter trait
  • stable_hash_logical_block() — splitmix64-based deterministic hash

Unit tests cover BG routing stability, primary worker selection, u128 round-trips, and rejection of invalid inputs (bg_count=0, empty workers).

3. Error Types (curvine-common/src/error/fs_error.rs)

Three new error variants: HashBlockCacheMiss, HashBlockNotReady, HashBlockCorrupt. Added is_hash_block_read_retryable() — cache miss and not-ready are retryable, corrupt is not (but included for simplicity).

4. Physical Dataset (curvine-server/src/worker/storage/physical_dataset.rs — NEW)

Replaces VfsDataset. Owns DirList, SPDK metadata store, capacity counters. All methods on &self, shared as Arc. The old Dataset trait and vfs_dataset.rs (576 lines) are deleted. DirList::choose_dir now takes &self (chooser behind Mutex).

5. Regular Block Catalog & Store (curvine-server/src/worker/catalog/ — NEW, block/regular_store.rs — NEW)

RegularBlockCatalog is a plain HashMap<i64, BlockMeta> + atomic delete counter — no IO, no capacity state. RegularBlockStore orchestrates the catalog with the physical dataset: open/finalize/abort/remove, capacity accounting, SPDK lifecycle. Locking is RwLock on the catalog only.

6. BlockStore Facade & Metrics (block/block_store.rs, worker_metrics.rs)

BlockStore now holds Arc<PhysicalDataset> + Arc<RegularBlockStore> and delegates. BlockStoreStats struct replaces the old Dataset trait reads. WorkerMetrics::text_output() uses the new stats() method instead of acquiring a read lock.

7. Design Docs (root-level .md files)

curvine-hashbg-file-read-acceleration-design.md and curvine-worker-block-layering-design.md document the overall architecture and the P1–P6 phased plan.
