1. Background

A storage refactor is easiest to understand if we first separate two ideas that often get bundled together: the physical place where bytes live, and the logical catalog that says which block owns those bytes.

1.1 The beginner's picture: Master, Worker, and blocks

Curvine splits files into blocks. The Master is the metadata and scheduling authority. Workers store the actual bytes. When a client writes a file block, the broad story is:

Regular block write flow
Clientwants to write block 100
→
Masterallocates block and target worker
→
Worker RPCreceives bytes
→
Local storagefile-backed dir or SPDK bdev
→
Heartbeatreports storage and block state

The worker is therefore both a byte store and a reporting agent. It must know where bytes are placed, how much capacity is used, which storage directories have failed, and which block ids are currently present.

1.2 Data dirs and SPDK

A worker has configured data dirs. A data dir may be a normal file-backed directory such as MEM, SSD, HDD, or DISK. It may also be StorageType::SpdkDisk, which is very different: the block is not a normal file, but a byte range on an SPDK bdev.

File-backed block

  • Has a local path such as active/.../blk_100.
  • Deletion can unlink a file.
  • Length can be checked with filesystem metadata.

SPDK block

  • Has a bdev name and byte offset.
  • Deletion releases an offset range.
  • Metadata must be persisted in SpdkMetaStore.

The common runtime record for a regular worker block is BlockMeta. It records the block id, length, state, owning directory, actual length, and optional SPDK offset.

BlockMeta {
    id: 100,
    len: 67108864,
    state: BlockState::Finalized,
    dir: Arc<DirState>,
    actual_len: 67108864,
    bdev_offset: -1,
}

1.3 The narrow background: why the old Dataset shape became awkward

Before this branch, the worker storage model centered on a Dataset trait and a VfsDataset implementation. That object carried two responsibilities at once.

Old shape: one object with two meanings
BlockStore holds Arc<RwLock<BlockDataset>>
VfsDataset as physical resource manager: data dirs, SPDK metadata, recovery, capacity
VfsDataset as logical catalog: HashMap<i64, BlockMeta>, open/finalize/abort/remove regular blocks

This is workable for regular blocks, because a regular block is keyed by an i64 block id. HashBG read cache needs a different key: purpose, BG id, logical file block, and block version. Extending the old dataset would force a regular-block catalog to pretend it also understands HashBG.

Important distinction. P1 is not the branch that makes workers read or write HashBG cache. It creates the foundation: common contracts plus a worker storage split that makes such support possible later.

2.1 A toy regular-block example

Suppose the client writes regular block 100, expected length 64 MiB. The old implementation did the physical allocation and the logical catalog mutation inside VfsDataset. The new implementation sends the same operation through clearer components.

New regular block path
BlockStore.open_blockstable facade API
→
RegularBlockStoreorchestrates lifecycle
→
PhysicalDatasetchooses dir / SPDK range
→
RegularBlockCatalogrecords 100 → BlockMeta

The behavior remains regular-block behavior. The improvement is that later HashBG work can introduce a separate catalog without bending the regular-block map into a different shape.

2.2 A toy HashBG example

Imagine a file with inode 42. A read at offset 128 MiB with 64 MiB blocks maps to file block index 2. If the file version is 7, the logical id is:

LogicalBlockId {
    inode_id: 42,
    block_index: 2,
    f_version: 7,
}

The future HashBG key is not a simple i64. It includes purpose and BG routing information:

HashBlockKey {
    purpose: BlockPurpose::ReadCache,
    bg_id: hash(42, 2) % bg_count,
    block_id: LogicalBlockId { inode_id: 42, block_index: 2, f_version: 7 },
    block_version: 0,
}
The key intuition. P1 creates room for two logical worlds to share the same physical worker resources: regular Curvine blocks and future HashBG cache blocks.

3. Code walkthrough

The branch is best read in five groups: common contracts, physical storage split, regular block catalog/store, BlockStore facade, and observability adaptation.

3.1 Common contracts: proto, topology, errors, RPC codes

common.proto

wire contract

Adds LogicalBlockIdProto with inode_id, block_index, and f_version.

worker.proto

future RPC payloads

Adds Hash block write/read/delete request and response messages.

topology/cluster_view.rs

routing model

Adds LogicalBlockId, HashBlockKey, BgPlacement, HashBgRouter, and ClusterView.

fs_error.rs + rpc_code.rs

client semantics

Adds HashBG-specific read errors and reserved RPC codes 84, 85, 86.

RpcCode::WriteHashBlock = 84
RpcCode::ReadHashBlock = 85
RpcCode::DeleteHashBlock = 86

The important thing is that these are contracts. P1 does not yet connect them to worker handlers.

3.2 Worker storage refactor: from VfsDataset to PhysicalDataset

The old Dataset trait and VfsDataset implementation are removed. Their physical-resource responsibilities move into PhysicalDataset.

PhysicalDataset owns physical resources, not logical block identity
PhysicalDatasetdata dirs, SPDK meta, capacity, recovery
↔
DirList / VfsDirchoose dir, check health, reserve bytes
↔
SPDK metadatabdev offset records

One subtle change is in DirList::choose_dir. It now takes &self, and the mutable round-robin state moves behind a Mutex<ChoosingPolicy>. That lets PhysicalDataset be shared by Arc while still preserving choice state.

3.3 RegularBlockStore and RegularBlockCatalog

Regular block identity moves into a dedicated catalog:

pub struct RegularBlockCatalog {
    blocks: HashMap<i64, BlockMeta>,
    num_blocks_to_delete: AtomicUsize,
}

The lifecycle logic moves into RegularBlockStore:

pub struct RegularBlockStore {
    dataset: Arc<PhysicalDataset>,
    catalog: RwLock<RegularBlockCatalog>,
}

This store orchestrates open, finalize, abort, remove, async remove, and recovery-backed initialization. It is the place where catalog mutation and physical operations meet.

3.4 BlockStore becomes a facade

External worker code still talks to BlockStore, but the internals change from one locked dataset to two dedicated collaborators.

pub struct BlockStore {
    dataset: Arc<PhysicalDataset>,
    regular: Arc<RegularBlockStore>,
}

That facade keeps existing regular-block APIs stable while giving future PRs a natural place to attach HashBgBlockStore.

3.5 Heartbeat and metrics stop peeking into Dataset

Heartbeat and metrics previously depended on direct dataset access. This branch introduces BlockStoreStats and BlockStore::stats(), so observability code reads an aggregate view instead of knowing the storage implementation.

pub struct BlockStoreStats {
    pub capacity: i64,
    pub available: i64,
    pub fs_used: i64,
    pub num_blocks: usize,
    pub num_blocks_to_delete: usize,
    pub failed_dirs: usize,
    pub total_dirs: usize,
}

This is a small but important boundary: once there are multiple catalogs, metrics can still be produced through a facade-level snapshot.

4. Review risks

This branch is mostly structural. The main risk is not that it implements the wrong HashBG behavior; it is that it accidentally changes regular-block behavior while making room for HashBG.

  • Regular block equivalence. Review open, finalize, abort, remove, async remove, startup recovery, SPDK metadata updates, and capacity accounting for behavioral parity with the deleted VfsDataset.
  • Recovery must not double count. PhysicalDataset::recover_blocks() feeds RegularBlockStore initialization. Future stores must not call it in ways that duplicate accounting.
  • Locking changed. The old coarse Arc<RwLock<VfsDataset>> becomes catalog locks, a choosing-policy mutex, and atomic counters. Check mutation boundaries carefully.
  • Pending delete counter semantics. num_blocks_to_delete moved into RegularBlockCatalog. Missing-block paths should be reviewed for counter balance.
  • Contracts are not handlers. The branch declares HashBG proto messages and RPC codes, but worker handlers still do not process Hash block RPCs.
  • Static topology is scaffolding. ClusterView::static_three_workers() is useful for the early stack, but should not be mistaken for production topology delivery.
  • Hashing intentionally ignores f_version. BG routing uses inode and block index, not file version, so different file versions land in the same BG.
  • HashBlockCorrupt is retryable. That is sensible for cache replicas, but reviewers should confirm it matches later fallback semantics.
  • HashBG placement excludes SPDK. hash_bg_placement selects file-backed dirs only in phase 1.
